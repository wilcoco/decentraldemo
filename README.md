# 아고라 라이브 (Agora Live)

**투표를 대체하는 실시간 민주주의 의사결정 시스템** — 데모 구현

## 문제의식

임기제 선출 투표는 "왕을 투표로 뽑는" 것과 같은 구조적 문제를 가진다.
선출 시점의 여론이 임기 내내 고정되는 것은 시스템에 **레그(lag)** 를 심는 것이며,
이 레그는 포퓰리즘·책임 회피·민의와 권력의 괴리를 낳는다.

이 프로젝트는 다음 원칙으로 레그를 제거한다:

1. **임기 없음** — 지지는 언제든 즉시 철회할 수 있고, 모든 의견의 지위는
   저장되지 않고 매 순간 현재 지지·검증 상태로부터 다시 계산된다.
2. **과제 분할** — 국정 과제를 분야(복지·경제·환경 등)별 의제로 분할해,
   시민이 사안 단위로 참여한다. 단일 대표에게 모든 권한을 몰아주지 않는다.
3. **의견의 발현과 검증** — 단순 찬반 투표가 아니다. 시민이 사안에 대해
   다양한 의견(주장)을 제안하고, 좋은 의견은 실시간 지지를 얻어 부상하며,
   근거 제출과 반론 제기를 통해 다양한 주체에 의해 검증된다.
   반론이 우세한 의견은 지지가 높아도 채택되지 않는다.
4. **분야별 위임 (액체 민주주의)** — 직접 참여하지 않는 분야는 신뢰하는
   시민에게 목소리를 위임할 수 있다. 위임은 연쇄적으로 흐르고(순환은 안전하게
   차단), **직접 참여가 항상 위임을 우선**하며, 위임은 즉시 회수할 수 있다.
5. **신뢰 네트워크 (자체 블록체인)** — 모든 제안·지지·철회·위임·검증 행위는
   시민의 ECDSA 개인키로 서명되어 해시 체인 블록에 기록된다.
   누구나 체인 전체(해시 연결·작업 증명·서명)를 재검증할 수 있어,
   조작 불가능한 감사 기록이 된다.

## 의견의 실시간 지위

| 지위 | 조건 |
|------|------|
| 제안됨 | 검증 활동 없음 |
| 검증중 | 근거 또는 반론이 존재 |
| 우세 | 유효 지지 가중치가 전체 시민의 25% 이상 |
| **채택** | 유효 지지 50% 이상 **그리고** 검증됨(근거 ≥ 반론) |
| 반박됨 | 반론이 근거보다 많음 (지지와 무관하게 채택 불가) |

지위는 어디에도 저장되지 않는다. 지지가 빠지면 그 순간 채택 지위도 사라진다.
이것이 "레그 없는 민주주의"의 핵심이다.

## 실행

의존성이 전혀 없다. Node.js 18+ 만 있으면 된다.

```bash
npm start        # http://localhost:3000
npm test         # 핵심 로직 테스트
```

웹 UI에서:
- 시민을 선택(또는 새로 등록)하고 의제별로 의견 제안 / 지지·철회 / 근거·반론 제출
- 분야별 위임을 설정·회수 (오른쪽 패널)
- **시뮬레이션 시작** 버튼을 누르면 가상 시민들의 활동으로 여론이 실시간으로
  움직이는 것을 볼 수 있다 (SSE로 모든 접속자에게 즉시 반영)
- 오른쪽 하단에서 신뢰 체인의 최근 블록과 무결성 상태를 확인

## 아키텍처

```
src/blockchain.js   신뢰 네트워크: ECDSA 서명 트랜잭션 + 해시 체인 + 작업 증명 + 전체 검증
src/democracy.js    핵심 로직: 시민/의제/의견, 실시간 지지, 위임 가중치 해석, 지위 계산
src/seed.js         데모 초기 데이터 (국정 과제 6건, 시민 12명, 의견·위임·검증 활동)
src/server.js       의존성 없는 HTTP 서버: JSON API + SSE 실시간 스트림 + 시뮬레이션
public/             한국어 웹 UI (실시간 갱신)
test/test.js        체인 무결성·변조 감지·위임 해석·지위 변동 테스트
```

### 위임 가중치 해석 규칙

의제 단위로 계산한다:
- 해당 의제의 어떤 의견에든 **직접 지지한 시민은 자기 목소리를 쓴다** (가중치 1).
- 직접 행동하지 않은 시민의 표는 그 의제 분야의 위임 사슬을 따라 흐른다.
- 사슬의 끝이 직접 지지자면 그의 가중치에 +1, 아니면 (미참여·순환) 그 표는 계산에서 제외.
- 의견의 유효 지지 = 그 의견의 직접 지지자들의 가중치 합.

## 2단계: 블록체인을 넘어 — 위브(Weave) 프로토콜

`src/weave/`는 분권화의 다음 단계다. 전역 블록체인을 버리고
**"시민별 서명 로그 + 관심 기반 복제 + 상호참조"** 구조를 택했다.

### 왜 블록체인이 아닌가

블록체인의 경쟁적 체인 연장(채굴/합의)은 **이중지불을 막기 위한 전역 단일
순서**가 필요해서 존재한다. 그러나 의견 표명에는 이중지불 문제가 없다 —
필요한 것은 "각 시민의 최신 상태"뿐이다. 따라서:

- **전역 체인 없음**: 시민마다 자기만의 서명 로그(개인 해시 체인)를 가진다.
  행위 1건의 비용 = 서명 1개. 채굴도 수수료도 없다.
- **전역 합의 없음**: 집계는 (작성자, 순번) 기준 최신 우선 병합(CRDT)이라
  항목이 어떤 순서로 도착해도 같은 결과에 수렴한다. 어느 노드가 계산해도 동일하다.
- **전체 복제 강제 없음**: 각 노드는 자기가 관심 있는 주제만 복제·저장한다.
  모든 사람이 신뢰 네트워크 유지에 리소스를 쏟을 필요가 없다.

### 신뢰의 세 기둥 (경쟁적 합의의 대체물)

| 위협 | 방어 |
|------|------|
| 위조 (남의 명의로 기록) | **서명** — 항목 전체가 시민 개인키로 서명됨 |
| 조작 (저장된 항목 고쳐 쓰기) | **서명** — 한 글자만 바꿔도 검증 실패 |
| 검열 (불리한 항목 몰래 삭제) | **상호참조 체크포인트** — 노드들이 주제 상태 요약을 서로의 로그에 엮어 두므로 대조 시 적발 |
| 이중 발언 (진영마다 다른 서명) | **분기 증명** — 같은 순번의 두 서명이 만나는 순간 암호학적 증거가 되고 집계에서 제외 |

핵심 성질: **관심이 곧 보안이다.** 어떤 주제에 관심 있는 사람이 많을수록
복제본과 상호참조가 늘어나 그 주제의 기록은 더 조작하기 어려워진다.
중요한 의제일수록 저절로 더 안전해지는, 민주주의와 정렬된 보안 모델이다.

```bash
npm run demo:weave   # 4개 노드 네트워크: 부분 복제 → 가십 수렴 → 3가지 공격 탐지 시연
```

```
src/weave/entry.js  시민별 서명 로그 항목 + 지갑(개인키는 시민 단말에만 존재)
src/weave/node.js   위브 노드: 관심 기반 복제, 가십 동기화, 체크포인트 상호참조, 결정적 집계
demo/weave-demo.js  분권 시나리오 데모 (수렴·이중발언·조작·검열 탐지)
test/weave.test.js  프로토콜 테스트 8건
```

## 3단계: 지지의 줄서기 구조 — 참여가 곧 검증

`src/weave/queue.js`는 지지를 **"그 의견 뒤에 줄을 서는 것"** 으로 재정의한다.

- **줄서기(JOIN)** = "나는 이 항목(들) 뒤에 선다"를 서명하는 것. 서명이
  앞사람 항목의 해시를 덮으므로, **줄에 서는 행위 자체가 앞사람들의 증인이
  되는 행위**다. 내 뒤에 누군가 서는 순간 내 자리는 지워질 수 없게 고정된다.
  검증 노동이 따로 필요 없다 — 참여가 곧 검증이다.
- **줄의 길이 = 현재 서 있는 사람 수.** 링크는 지울 수 없는 역사로 남고,
  떠나기(LEAVE)는 길이만 줄인다. 실시간 철회 원칙이 그대로 유지된다.
- **갈라진 줄은 저절로 아문다.** 두 사람이 동시에 줄 끝에 서면 줄이 갈라지는데
  (분산 시스템의 자연 현상), 다음 사람이 갈라진 끝들을 모두 참조하며 서면
  줄이 병합된다. 사람 수는 중복 없이 센다.
- **분기(AMEND) = 의견의 진화.** 동의하되 수정하고 싶으면 줄의 한 지점에서
  갈라져 수정안의 새 줄을 시작한다. 의견은 나무(변형 계보)로 진화하고,
  시민의 위치는 가족(원안+수정안들) 안에서 하나뿐이다 — 수정안 줄로 옮겨
  서면 원안 줄에서 자동으로 빠진다. 어떤 변형이 가장 긴 줄을 얻는지가
  실시간으로 드러난다.
- **검열 방어가 지지에 비례한다.** 줄 중간의 항목을 몰래 지우면 그 뒤에 선
  모든 사람의 참조가 허공에 뜬다 — 뒷사람 전원이 증인이다.
  **긴 줄일수록 지우기 어렵다. 지지의 크기가 곧 기록의 견고함이다.**

```bash
npm run demo:queue   # 줄서기 → 동시성 치유 → 분기 → 떠나기 → 검열 적발 시연
```

## 4단계: 안목 지수 — 링크 순서 기반 평가 계층 (특허 기반)

`src/weave/insight.js`는 **등록특허 10-0913256**(다중 링크를 형성하는 정보
네트워크에서 링크 관계에 따른 대상 평가 방법)과 **등록특허 10-0952391**(콘텐츠
평가에 따른 가치 분석 시스템)의 핵심 아이디어를 줄서기 구조 위에 구현한 것이다.

> **권리 상태**: 두 특허 모두 출원일(2005-04-14)로부터 20년의 존속기간이
> 2025-04-14에 만료되어 현재 **누구나 자유롭게 실시할 수 있다** (공공재).
> 발명자 홍정수의 발명자 지위(공보 게재)는 영구적이다.
> 상세 요약은 `docs/patents.md` 참조.

### 특허의 핵심 (2005, 발명자 홍정수)

- 평가자(상위 노드)의 **허브 지수는 같은 대상에 "이후로 링크한" 다른 평가자에
  의해 결정**된다 — 좋은 대상을 일찍 알아본 사람일수록, 나중에 따라오는
  사람이 많을수록 안목이 실시간으로 올라간다 (청구항 1·2).
- 대상의 **권위 지수 = 링크한 평가자들의 허브 지수 합** (수학식 10).
- 여러 대상에 대한 허브 지수는 **평균**으로 산출한다 (청구항 8) —
  아무도 따라오지 않는 대상에 무분별하게 링크하면 평균이 깎이는 자기 조절.
- 클라인버그/페이지랭크와 달리 **반복 수렴 연산이 불필요**해 실시간 갱신 가능.

### 줄서기 구조와의 결합

특허의 "링크 순서"는 서버 타임스탬프에 의존했지만, 줄서기 DAG에서는 순서가
서명 사슬 자체로 증명된다 — 뒤에 선 사람의 서명이 앞사람 항목의 해시를
덮으므로 "내가 먼저 알아봤다"는 주장이 위조 불가능한 사실이 된다.
**특허의 허브 지수가 암호학적으로 검증 가능한 평판이 되는 것이다.**

| 지표 | 정의 | 의미 |
|------|------|------|
| 안목 지수 (줄별) | 내 뒤에 선 사람 수 (DAG 후손) | 그 사안을 알아본 순서 |
| 시민 안목 지수 | 참여한 줄들의 평균 | 좋은 의견을 일찍 알아보는 능력 |
| 의견 권위 지수 | 서 있는 시민들의 (1+안목) 합 | 줄 길이가 같아도 안목 있는 줄이 무겁다 |

민주주의 적용에서 모든 시민의 목소리는 기본 1이고(1인 1표), 안목은 그 위에
가중을 더한다. 포퓰리즘(길지만 안목 없는 줄)과 숙련된 판단(안목 있는 시민들이
선 줄)이 지표로 구분된다.

```bash
npm run demo:insight   # 안목 축적 → 자기 조절 → 권위 지수 비교 시연
```

## 5단계: P2P 클라이언트 — 중앙 서버의 완전한 제거

`src/weave/peer.js`와 `bin/peer.js`는 위브 노드를 실제 네트워크 클라이언트로
만든다. **모든 피어는 서버이자 클라이언트다** — 중앙 서버가 없다.

- **부트스트랩**: 아는 피어(시드) 하나에만 접속하면 PEERS 교환(피어 발견)으로
  그물망이 스스로 자란다.
- **전파**: 내 행위는 내 기기에서 서명되어(개인키는 내 기기에만 존재) 이웃에게
  즉시 밀어 보내진다.
- **수렴**: 주기적 반보정(anti-entropy) 가십으로 "내가 가진 항목 목록"을
  교환해 서로 빠진 것을 채운다. 늦게 합류한 피어도 과거 전체를 따라잡는다.
- **장애 허용**: 피어가 죽어도 남은 피어끼리 계속 동작한다. 단일 장애점이 없다.
- **이중 발언 자동 적발**: 항목 목록이 해시까지 비교되므로 같은 순번의 상충
  서명이 반보정 과정에서 반드시 만나 분기 증명이 네트워크 전체에 퍼진다.
- 전송은 TCP + NDJSON이지만 프로토콜은 전송 독립적이다 — 같은 로직이
  WebSocket/WebRTC(브라우저) 위에도 얹힌다.

```bash
npm run demo:p2p     # 4피어 그물망: 발견 → 수렴 → 피어 사망 → 늦은 합류 시연

# 터미널 여러 개에서 직접 P2P 네트워크 만들기:
node bin/peer.js --name 김하늘 --port 4001 --topics t_연금개혁
node bin/peer.js --name 이도윤 --port 4002 --seeds 127.0.0.1:4001 --topics t_연금개혁
# 명령: p 제안 / j 줄서기 / l 떠나기 / a 수정안 / s 상태 / h 안목 / n 피어
```

신원은 데모에서 개방 등록(HELLO 시 공개키 자동 등록)이며, 실제 시스템에서는
이 지점에 DID + 영지식 자격증명 검증("실존 유권자의 키인가")이 들어간다.

### 전체 이슈 조회 — 카탈로그(목차) 주제

각 피어는 관심 주제만 복제하는데, 그러면 "네트워크에 어떤 이슈가 존재하는가"는
어떻게 아는가? 답은 **모두가 기본으로 복제하는 예약 주제 `t_@catalog`** 다.

- **"본문은 관심 있는 것만, 목차는 모두가."** 카탈로그에는 이슈의 공표
  항목(제목·설명·분야·topicId)만 담기므로 가볍다.
- **공표는 일반 줄서기 항목이다.** 이슈에 관심을 표명하는 것 = 공표 항목
  줄에 서는 것. 따라서 이슈별 관심도가 실시간 집계되고, 중요한 이슈를 일찍
  알아본 시민은 **의제 설정 단계에서도 안목 지수를 얻는다.**
- **구독은 즉시 백필된다.** `follow(topicId)`로 관심사를 넓히면 갱신된
  관심사가 이웃에게 재공지되고, 반보정 가십이 그 주제의 과거 항목 전체를
  채워 준다.
- 카탈로그도 위브 주제이므로 서명·분기 증명·상호참조가 그대로 적용된다.
- 규모가 커지면 분야별 카탈로그 샤딩(`t_@catalog/복지` 등)이나 DHT 기반
  발견으로 확장한다.

### 키워드 검색 — 두 층위

- **이슈(목차) 검색은 로컬에서 즉시 완결된다.** 카탈로그는 전원 복제이므로
  제목·설명·분야에 대한 키워드 검색에 네트워크가 필요 없다.
- **본문(의견) 검색은 질의 전파(query flooding)다.** 내가 복제하지 않는 주제
  속 의견은 질의를 이웃에게 흘려보내면(TTL 홉 제한), 그 주제를 복제 중인
  피어가 자기 로컬을 뒤져 결과를 역경로로 돌려준다. 같은 질의는 피어당 한
  번만 처리되고(qid 중복 제거), 같은 결과가 여러 경로로 와도 한 번만 집계된다.
- **"관심 있는 사람이 저장하고, 저장한 사람이 검색해 준다."** 저장 구조와
  검색 구조가 일치하므로 별도의 검색 서버가 필요 없다. 규모 확장 시에는
  DHT 역색인(키워드 → 주제 포인터)으로 대체한다.

```
김하늘> c 연금 개혁            (이슈 생성·공표)
김하늘> p 자동조정장치 도입     (의견 제안)
이도윤> k 자동조정              (미구독 상태에서 검색)
  - [의견] 자동조정장치 도입 — 이슈 "연금 개혁", 찾은 피어: 김하늘 (f 0 로 구독)
```

## 6단계: 입장 표명(지지/반대/의견 첨부)과 보안 계층

### 지지·반대·의견 첨부

의견마다 **지지 줄(JOIN)과 반대 줄(OPPOSE)** 이 따로 선다. 둘 다 같은
줄서기 구조이므로 증인·안목·무결성 장치가 대칭으로 적용된다.

- 지지/반대에 **자기 의견을 첨부**할 수 있고, 이것이 그 의견의
  지지의견·반대의견 목록으로 축적된다 (링크는 역사이므로 목록도 역사다).
- **가족 내 입장은 하나**: 지지에서 반대로 옮기면 지지 줄에서 자동으로 빠진다.
- 상태는 실시간 재계산: 제안됨 / 경합 / 우세 / 반대 우세.
- 검색 결과에 지지·반대·권위(양쪽)·지지의견/반대의견 수·상태가 담겨 온다.
- **헛발질 감점**: 내가 선 쪽이 명백히 열세면(반대가 지지를 초과한 격차)
  그만큼 안목이 깎인다 — 지금도 그 줄에 서 있는 동안만. 판단을 철회하면
  감점은 사라지고, 여론이 회복돼도 사라진다(레그 없음). 안목이 음수여도
  **기본 1표는 불가침**이다.
- CLI: `j <번호> [의견]` 지지, `o <번호> [의견]` 반대, `d <번호>` 상세
  (지지의견/반대의견 목록), `s` 상태에 지지/반대/권위 병기.

### 보안 계층

- **시빌 방어**: 자격증명 승인 모드 — 신뢰 발급자(`CredentialIssuer`)가
  서명한 공개키만 등록부에 진입. 무한 계정 생성으로 여론에 진입 불가.
- **스팸/DoS 방어**: 작성자별 속도 제한(기본 60건/분), 항목 데이터 8KB 상한,
  수신 메시지 512KB 상한(초과 시 회선 차단).
- 위협 모델 전체와 지수 파밍 분석, 적대적 행위의 해석·용인 기준은
  **`docs/security.md`** 참조.

## 7단계: 이론 로드맵 반영 — 선별적 마찰의 설계

`docs/theory-review.md`의 처방 8가지를 모두 구현했다. 핵심 철학:
**실시간을 죽이지 않고 선별적 마찰을 설계한다.** 마찰은 레그가 아니라
필터다 — 임기제는 여론이 바뀌어도 4년을 기다리게 하지만, 필터는 여론이
진짜 바뀌면 N일 뒤 반드시 반영한다.

| 처방 (이론 근거) | 구현 |
|---|---|
| 블라인드 초기 구간 (콩도르세: 판단의 독립성) | 공표 후 `blindMs` 동안 집계 비공개 플래그 — 논증만 보고 판단. 기록은 쌓이므로 안목 계산은 온전 |
| 지속 다수 (매디슨: 순간적 격정의 여과) | 채택 = 임계값을 `sustainMs` 동안 유지. 조건이 깨지면 시계 리셋. 서명된 타임스탬프 재생으로 결정적 계산 |
| 헌장 계층 (토크빌: 다수의 폭정 방어) | 헌장 의제(`cc`)는 채택 기준 2/3 + 3배 유지 기간. 공표 항목이 불변이므로 헌장 표시는 사후에 뗄 수 없음 |
| 무작위 검증 배심 (피시킨·랑드모어: 동원 불가능성) | 의견 해시로 결정론적 추첨(누구나 재계산해 검증 가능, 제안자 제외). 배심 다수 승인 없이 채택 불가, 다수 기각 시 지지가 높아도 기각 — 하버마스의 숙의 관문 겸용 |
| 안목 상한·시간 감쇠 (홍-페이지·미헬스) | 권위 기여 상한(기본 cap=2), 반감기 감쇠 옵션 — 과거의 영광이 종신 권력이 되지 않도록 |
| 다양성 지표 (랑드모어: 다양성>능력) | 지지자들의 입장 프로필 유사도의 보수 — "같은 무리가 또 선 줄"과 "다양한 연합의 줄"이 수치로 구분 |
| 교차 노출 (선스타인: 에코 체임버) | 지지 직전 최신 반대의견을 먼저 표시 |
| 위임 (다운스: 합리적 무지) + 지역 연방화 (오스트롬) | 주제 단위 위임(`w`) — 사슬로 흐르고, 직접 참여 우선, 순환 차단, 즉시 회수. 이슈에 지역·헌장 표시 |

```bash
# 거버넌스 파라미터를 켠 클라이언트:
node bin/peer.js --name 김하늘 --blind 60 --sustain 1440 --jury 5
#   --blind 60    : 공표 후 60분 집계 비공개
#   --sustain 1440: 채택 조건 24시간 유지 필요 (헌장 의제는 3배)
#   --jury 5      : 의견마다 5인 추첨 배심의 다수 승인 필요
```

## 8단계: 브라우저가 곧 피어다 — 웹 P2P 클라이언트

`bin/signal.js` + `public/p2p/`. **웹페이지를 여는 순간 그 브라우저가 완전한
피어가 된다.** 설치도, 계정 서버도 없다.

```bash
npm run web            # 신호 서버 실행 (기본 8080)
# 브라우저 여러 개(다른 기기·탭)에서 http://<호스트>:8080 접속
# 거버넌스 파라미터: http://호스트:8080/?blind=60&sustain=300&jury=3 (초/명)
```

- **키는 브라우저 안에서 태어나 브라우저를 떠나지 않는다** — WebCrypto로
  생성, localStorage에 보관(데모), 새로고침해도 신원이 유지된다.
- **신호 서버는 전화번호부이지 전화국이 아니다** — WebRTC 연결 성립까지만
  중개하고, 이후 모든 항목(제안·지지·반대·위임)은 브라우저 사이의 데이터
  채널로 직접 흐른다. 서버는 누가 무엇을 지지하는지 볼 수 없고, 죽어도
  연결된 피어들은 계속 동작한다.
- **집계 코드는 서버 피어와 동일한 모듈** — 브라우저가 `/src/weave/queue.js`
  `insight.js`를 그대로 import 하므로, 배심 추첨·지위·안목·다양성 계산이
  Node 피어와 한 글자도 다르지 않다 (이를 위해 해시를 순수 JS로 분리:
  `src/weave/hash.js`, 상수: `consts.js`).
- **메모리를 잃어도 네트워크가 기억한다** — 새로고침으로 항목을 잃은 피어는
  이웃의 반보정 가십이 역사 전체를 복원해 준다 (E2E로 검증).
- UI에 이론 로드맵 전체가 반영됨: 블라인드 카드(집계 숨김), 헌장 배지,
  지속 확인 표시, 배심 패널(내가 배심원이면 승인/기각 버튼), 교차 노출
  (지지 직전 반대의견 확인 대화상자), 위임 선택, 다양성·권위 병기.
- 한계(데모): 신원은 개방 등록(자격증명 검증은 Node 피어와 동일 지점에
  이식 가능), 키는 localStorage(실전: IndexedDB 비추출 키 + 소셜 복구),
  NAT 뒤 기기 간에는 TURN 중계 필요, 브라우저망과 TCP 피어망의 서명
  형식 통일(DER↔raw)은 다음 단계.

이로써 1단계의 중앙 서버 데모(`npm start`)는 역사적 참고용이 되었다 —
같은 화면 경험이 이제 중앙 서버 없이 동작한다.

### 서비스 최초 접근 — 안내 홈 + 접속 즉시 참여

배포하면 `/`가 취지를 설명하는 안내 홈페이지이고, "지금 참여하기" 버튼이
`/app`(브라우저 피어)으로 이어진다. **다운로드가 없다** — 브라우저가 곧
피어이기 때문이다. 상시 노드를 원하는 사람을 위한 CLI 안내는 홈에 포함.

Railway 배포: 저장소를 연결하면 `railway.json`/`Procfile`이
`node bin/signal.js`를 실행한다 (PORT 환경변수 자동 인식, HTTPS/WSS는
Railway가 제공). 주의: 신호 서버는 부트스트랩 지점일 뿐 데이터를 보지
못하지만, 접근성의 단일 지점이므로 장기적으로는 대체 신호 서버 목록을
공표해 두는 것이 좋다. 서로 다른 통신사 NAT 뒤 기기 간에는 TURN 중계
(coturn 등)를 추가해야 연결률이 올라간다.

## 데모의 한계 (실제 시스템으로 가려면)

- 개인키가 서버에 보관된다 → 실제로는 클라이언트(시민 단말)가 키를 보관하고 서명해야 한다.
- 단일 노드 체인이다 → 실제로는 합의 알고리즘(PoS/BFT)을 갖춘 분산 노드가 필요하다.
- 신원 검증이 없다 → 1인 1목소리를 위한 주민 인증(영지식 증명 등) 연동이 필요하다.
- 메모리 저장이다 → 영속 저장소가 필요하다.
- 어뷰징 방어(시빌 공격, 여론 조작 봇)와 숙의 품질 장치(토론 단계, 냉각 기간 옵션)가 필요하다.
